**Status:** Pure inventory phase. **No fixes proposed.** No code modified.

---

## How to read this

- The table is sorted by **CONFIDENCE, lowest first** — the most suspicious metrics are at the top.
- "Confidence" rates how much we trust that *the number shown reflects a real, correctly-attributed measurement* — not whether the code runs.
- "Derived" in the Source column means the value is computed from other numbers, not measured directly.
- ⚠️ in Known Issues flags a fabrication, contradiction, or stale-data hazard called out in the Findings Summary.

## Findings Summary

### Top 5 highest-risk metrics

1. **Gateway Security Score = hardcoded `80` (row 1).** `NWCONNECTION_TESTS_ENABLED = false` (GatewaySecurityScanner.swift:17) means the scanner never measures anything and returns the literal `80` (lines 43, 71). This constant then flows into the weighted **Overall Security Score** (12% weight), so the headline security verdict is partly synthetic.
2. **IP Reputation flags = hardcoded `false` (row 2).** `isBlacklisted/isSpam/isProxy` are literals (IPReputationScanner.swift:136–139); the HTTP threat APIs are ATS-blocked (comment line 129). Only Tor detection (DNS-based) is real. Every non-Tor user scores ~80–100 regardless of actual reputation.
3. **Trends "Down X%" cross-network contamination (row 15).** `TrendAnalyzer.analyzeSpeedTrends` (TrendAnalyzer.swift:38–61) averages the last-3 vs prior-3 speed tests with **no `connectionType` filter**, even though `SpeedTestResult` records it. A Wi-Fi→cellular switch is reported as a degradation. The `referenceLatencyMs` guard (lines 171–184) only filters `metric == "latency"` — download deltas like "Down 93%" are never suppressed.
4. **Internet card two-latency contradiction (rows 32–33).** The status line "Connected (98 ms)" reads **raw** `status.internet.latencyToExternal` (DashboardViewModel.swift:596–599) while the "Latency 342 ms" row reads the **5-sample smoothed** value (DashboardView.swift:616 → ViewModel:841–867). Two numbers, two pipelines, one card.
5. **VPN detection false-positive risk (row 14).** `SmartVPNDetector` (lines 353–416) declares "VPN Active" at confidence ≥0.80, which a datacenter-ISP classification alone (0.95) can reach. A wrong ASN lookup or a transient block of `google.com` (the China-locale path scores 0.70) can flip the verdict with no authoritative NEVPNManager confirmation.

### Contradictions detected (two metrics on the same screen disagree)

- **A1 — Internet card (rows 32, 33):** raw status latency vs smoothed Latency row. "Connected (98 ms)" beside "Latency 342 ms" is exactly this: the status string is the current reading, the row is the trailing 5-sample mean. They diverge on any volatile network.
- **A2 — Router card (rows 26, 57):** the **Status** line can render "Excellent" whenever *any* live gateway latency exists (DashboardView.swift:488 returns `NetworkHealth.excellent.displayName`), independent of magnitude — so "Status: Excellent" can sit directly above "Latency: 95 ms" colored orange/red.
- **Scale collision — "Excellent" is overloaded:** Header health "Excellent" = score ≥70 (ViewModel:940); Router status "Excellent" = *any* live latency (DashboardView:488); color-green "Excellent" = <30 ms (`forLatency`) or <10 ms (`forGatewayLatency`); RootCauseAnalyzer router-Excellent = <10 ms, internet-Excellent = <50 ms; VPN health "Excellent" = <30 ms overhead; DNS rating "Excellent" = <20 ms (benchmark) or <30 ms (dashboard color). The same word denotes at least six different scales.
- **Router latency color scale mismatch (row 57):** the dashboard router Latency row is colored by `forLatency` (<30 ms green) while the gateway has a dedicated tighter `forGatewayLatency` (<10 ms green) and the health/quality logic penalizes from >10/>30 ms — so a 25 ms gateway shows green yet costs health points.
- **Jitter label vs color (row 38):** SpeedTestView explanation text (682–691) calls 25–50 ms "moderate/high" while the color function (741–745) keeps everything <50 ms out of red and treats <25 ms as one green band; a 42 ms jitter is described and colored inconsistently.
- **DNS "latency" vs "resolution time" (rows 8, 45):** the dashboard DNS card shows a latency read from NetworkMonitorService (DNSAnalyzer.swift:267, not measured locally), while the DNS Benchmark measures a real UDP round-trip (DNSBenchmark.swift:341–402). Same word, different number, potentially shown in different tabs.

### Fabricated / derived values masquerading as measurements

- **Hardcoded constants:** Gateway Security Score `80` (row 1); IP-reputation blacklist/spam/proxy `false` (row 2); WiFi encryption type `false` (row 54).
- **Unmeasurable-on-iOS placeholders:** WiFi Saturation/airtime (row 3), Router MTU (row 4), NAT type (row 5), 802.11 roaming-protocol support (row 6) — iOS exposes no API for these; the scores are stubs or heuristics.
- **Derived presented as measured:** Streaming "VPN Impact %" is `withVPN × (1 + vpnLatency/100)` with no real VPN-off baseline (row 10); VPNModeBenchmark upload = `download × 0.3` (row 36/Finding F); Traceroute "hops" are HTTP RTT to fixed waypoints, not path routers (row 9).
- **N/A test that can still raise a threat:** WebRTC leak (row 7) — one path always passes, another can list it as a critical finding, impossible in a native app.

### Thresholds without a documented calibration source

Every verdict cutoff in the app is a **hand-tuned magic number**; none cites an RFC, ITU spec, ISP SLA, or measured percentile distribution. The most load-bearing un-calibrated thresholds:

- **Health Score** rubric & band cutoffs — DashboardViewModel.swift:873–962 / RootCauseAnalyzer.swift:706–779. Comments say "calibrated on real-world measurements" and give worked examples, but no external standard; inputs are HTTP-HEAD latencies (TLS-inflated), not ICMP (Finding E).
- **Security Score** sub-score weights (DNS/Gateway/TLS/Behavior 12% each, etc.) and the `−5 per threat` penalty — SecurityIntelligenceEngine.swift:1033–1088.
- **VPN scores** (Auto/Privacy/Failure/Reliability) — all additive hand-tuned weights, VPNAutoScorer/VPNPrivacyScore/VPNFailurePredictor/VPNReliabilityTracker.
- **Color scales** — NetworkColors.swift latency (21–28), gateway (37–44), speed (55–62), DNS (141–149), VPN overhead (159–166): reasonable but undocumented and internally inconsistent (three different DNS scales; two latency scales).
- **Trend trigger thresholds** ±20%/±30% (TrendAnalyzer:44/52/70/78) and the 50% suppression ratio (line 183).
- **Smart-recommendation** gateway>50 / DNS>300 / loss>2% / ext>100 (SmartRecommendationEngine:67–143) — DNS threshold was moved 50→300 ms purely to reduce noise (comment line 115).

### Stale-data risks (cached measurements that may mislead across network changes)

- **Trends across networks (rows 15–17):** speed/diagnostic history is pooled across Wi-Fi and cellular with no `connectionType` filter; old entries (the engine keeps up to 50 speed tests / 1000 history entries) drive "Down X%" claims long after conditions changed. This is the single largest stale/contamination hazard.
- **WiFi quality estimator (row 53):** live NWConnection probing is disabled (WiFiQualityEstimator.swift:72); "WiFi: Good" can reflect a latency reading minutes old.
- **Diagnostic root-cause override (row 60):** a diagnosis up to 10 minutes old (DashboardViewModel.swift:62–64) can override the live connection-quality string after the user has moved networks.
- **Smoothed latencies (rows 33, 57):** the 5-sample rolling mean persists across a network switch (the history arrays are never cleared on path change in DashboardViewModel), so the smoothed Internet/Router/DNS latency can show a blend of the old and new network for several refreshes — and is precisely what produces the A1 raw-vs-smoothed split.
- **GeoIP 5-minute cache (row 50):** acceptable, but on a fast Wi-Fi↔cellular flip the public IP / ISP / location can lag up to 5 minutes (GeoIPService cache).
- **Connection-stability avg latency (row 52):** pools latency from whatever interface was active, unlabeled, so the 24h "avg latency" can blend Wi-Fi gateway and cellular external latencies.

### Additional cross-cutting notes

- **Finding F — upload speed has two implementations:** `SpeedTestEngine.swift:329` and `VPNBenchmarkEngine.swift:213–236` perform a real 5–10 MB POST (HIGH confidence), but `VPNModeBenchmark.swift:185` returns `downloadSpeed × 0.3`. Commit a9c1864 ("stop fabricating upload speed from download") fixed one path; the ModeBenchmark derivation appears to remain. Which path renders depends on the caller — worth confirming in the fix phase.
- **Finding E — HTTP HEAD latency vs ICMP thresholds:** router/internet/DNS latencies that feed the health score are measured with `URLSession` HEAD/GET requests (TLS + TCP setup included), yet the thresholds (router "Excellent" <10 ms) are calibrated as if they were raw ICMP RTT. iOS forbids raw ICMP sockets, so the measurement floor is structurally higher than the threshold assumes.
- **Two parallel history stores:** `HistoryManager` (UserDefaults, SpeedTestResult, feeds TrendAnalyzer) and `HistoryStorageService`/`NetworkHistoryManager` (file-based, NetworkHistoryEntry, feeds HistoryInsightsEngine). VPN-impact analysis exists only in the latter, so the dashboard Trends card is blind to VPN state when computing speed deltas.
